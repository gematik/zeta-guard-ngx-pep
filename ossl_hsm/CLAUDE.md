# HSM Proxy Provider (ossl_hsm)

OpenSSL 3.x provider in Rust that proxies TLS private key operations to a gRPC HSM backend.
Part of the ZetaGuard product — enables users to secure nginx TLS keys with their own HSM by implementing a gRPC API.

## Architecture

```
nginx (TLS termination)
  │
  ▼
OpenSSL 3.x ──── STORE("hsm:my-key.p256") ──► hsm_store.rs (URI → KeyHandle)
  │                                                   │
  ├── KEYMGMT("EC") ◄────────────────────────────────┘
  │     KeyHandle { key_id } ── no key material, just an ID
  │     query_operation_name("EC" → "ECDSA")
  │     export: calls gRPC GetPublicKey for cert/key matching
  │
  └── SIGNATURE("ECDSA")
        digest_sign: hash locally (digest matched to curve), then gRPC Sign
        converts P1363 (raw R|S) → ASN.1 DER for TLS
                │
                ▼
        gRPC HSM Proxy (production: real HSM, testing: hsm_sim)
```

## Call Flow

### nginx startup (key loading)

1. **`OSSL_provider_init`** (`lib.rs`) — OpenSSL loads our `.so` per `openssl.cnf`.
   Registers three algorithm families: EC keymgmt, ECDSA signature, hsm store.

2. nginx reads `ssl_certificate_key "store:hsm:my-key.p256"`. OpenSSL strips `store:`,
   routes `hsm:my-key.p256` to our store:
   - **`store_open`** (`hsm_store.rs`) — parses URI, extracts `key_id = "my-key.p256"`
   - **`store_load`** (`hsm_store.rs`) — creates `KeyHandle { key_id }` (just an ID, no
     key material), passes it to OpenSSL via callback as `{TYPE=PKEY, DATA_TYPE="EC", REFERENCE=&KeyHandle}`
   - **`store_eof`** (`hsm_store.rs`) — returns 1 (done, only one object)
   - **`store_close`** (`hsm_store.rs`) — frees `StoreCtx`

3. OpenSSL matches DATA_TYPE="EC" to our keymgmt:
   - **`keymgmt_load`** (`hsm_ec.rs`) — receives the `KeyHandle` pointer from the store reference, returns it as the key object

4. OpenSSL verifies key matches certificate — triggers the **first gRPC call**:
   - **`keymgmt_export`** (`hsm_ec.rs`) → `grpc_client::get_public_key("my-key.p256")` → gRPC `GetPublicKey`
   - Server returns the public key (hsm_sim derives it from key_id via HKDF)
   - Provider parses PEM, returns uncompressed EC point + group name via callback
   - OpenSSL compares against `ssl_certificate` — match = startup succeeds

5. OpenSSL also queries key properties:
   - **`keymgmt_get_params`** (`hsm_ec.rs`) — returns curve-derived values (bits, security-bits, max-size, group), curve sourced from the cached `KeyMaterial`
   - **`keymgmt_query_operation_name`** (`hsm_ec.rs`) — maps keymgmt "EC" → signature "ECDSA"

6. nginx forks worker processes. Master is done with key loading.

### HTTPS request (TLS handshake signing)

1. **`signature_newctx`** (`hsm_ec.rs`) — allocates `SignCtx`
2. **`signature_digest_sign_init`** (`hsm_ec.rs`) — copies `key_id` from `KeyHandle`,
   resolves the `Curve` from cached `KeyMaterial`, picks the digest (caller-provided
   `mdname` if any, else `Curve::default_digest()`), and stashes all three on `SignCtx`
3. **`signature_digest_sign`** (`hsm_ec.rs`) — called **twice**:
   - First call: `sig=NULL` → returns `siglen = curve.max_sig_der_len()` (72/104/137 by curve)
   - Second call: the actual signing:
     1. Hash locally with the digest chosen at init time over TLS "to-be-signed" data
     2. gRPC `Sign(key_id, digest, algorithm=NONE)` — server signs pre-hashed digest, returns IEEE P1363 (R||S zero-padded; 64/96/128 bytes by curve)
     3. Provider converts P1363 → ASN.1 DER (what TLS expects)
     4. Copies DER into output buffer
4. **`signature_freectx`** (`hsm_ec.rs`) — frees `SignCtx`

### Fork safety (`grpc_client.rs`)

The gRPC client checks `std::process::id()` on every call. After nginx forks workers,
the PID changes, so the first gRPC call in a worker creates a fresh tokio runtime and
connection. The runtime uses `new_current_thread()` (not multi-thread) because spawned
tokio worker threads don't survive `fork()`.

## Build & Test

```bash
# Build everything
cargo build

# Run hsm_sim tests (no server needed — tests start their own)
cargo test -p hsm_sim

# Start gRPC server for manual testing
cargo run -p hsm_sim

# Test gRPC directly (server must be running)
grpcurl -plaintext '[::1]:50051' list
grpcurl -plaintext -d '{"key_id":"my-key.p256"}' \
  '[::1]:50051' gematik.zetaguard.hsmproxy.v1.HsmProxyService/GetPublicKey

# Start nginx (OPENSSL_CONF is set by direnv)
# ssl_certificate_key must use store:hsm:<key_id> URI
nginx/sbin/nginx

# Test TLS handshake
curl -k https://localhost:10000/

# Stop nginx
nginx/sbin/nginx -s stop
```

## Key Conventions

- **Key IDs are opaque to the gRPC API** — real HSMs use arbitrary identifiers. The provider determines the EC curve by parsing the public key returned from `GetPublicKey` (curve OID in the EC group), not from the key-id string.
- **hsm_sim convention only**: the simulator expects asymmetric key_ids ending in `.p256`/`.p384`/`.bp256`/`.bp384`/`.bp512` so it can derive a deterministic EC key for that curve via HKDF-SHA256. Symmetric keys can be any string (AES-256 derived via HKDF-SHA256). This is purely an hsm_sim implementation detail — it is not part of the protocol contract.
- **Certificates**: hsm_sim generates leaf certs on the fly, signed by `keys/ca.{key,crt}`. Fixed validity: notBefore = Unix epoch, notAfter = 2337-01-01. Certs are cached atomically on disk.
- **Logging**: use `eprintln!("[ossl_hsm] ...")` in provider code. The `log` crate macros produce no output in provider context (no logger backend).
- **gRPC address**: provider reads `HSM_PROXY_ADDR` env var, defaults to `http://[::1]:50051`.

## Design Decisions

- **keymgmt = "EC"**, signature = **"ECDSA"** — must use standard OpenSSL algorithm names.
- **`keymgmt_query_operation_name`** bridges "EC" keymgmt to "ECDSA" signature. Without it, OpenSSL looks for a signature named "EC" and fails.
- **STORE DATA_TYPE = "EC"** must match keymgmt registration.
- **Fork safety**: nginx forks workers. The gRPC client uses `tokio::runtime::Builder::new_current_thread()` (not multi_thread — worker threads don't survive fork) and PID-based reconnection.
- **Signature format**: gRPC server returns IEEE P1363 (raw R|S). Provider converts to ASN.1 DER for TLS.
- **Supported curves** (per gemSpec_Krypt A_28868): P-256, P-384 (MUSS) and brainpoolP{256,384,512}r1 (KÖNNEN). P-521 is intentionally *not* accepted — the spec says "andere Kurven SOLLEN NICHT verwendet werden". The curve is discovered lazily on first use by parsing the public key from `GetPublicKey` and reading the EC group OID; the result is cached on the `KeyHandle`. Per-curve constants (bits, security-bits, max-DER-size, group name, default digest) live on the `Curve` enum in `hsm_ec.rs`.
